# Foundation & Deploy Pipeline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A deployable Next.js 16 PWA skeleton with pinned tooling (mise), local Docker Postgres wired through Drizzle, Vitest/Playwright harnesses proven by real tests, green GitHub Actions CI, and production deploys on Vercel.

**Architecture:** Greenfield scaffold via create-next-app merged into this repo. All commands run through mise tasks so local, CI, and future contributors share pinned tool versions. Database access goes through a single Drizzle client module; tests prove each harness end-to-end before any feature code exists.

**Tech Stack:** Next.js 16 (App Router, TypeScript, Turbopack), Tailwind CSS v4 + shadcn/ui, Drizzle ORM + node-postgres, Postgres 17 (Docker locally / Neon in prod), Serwist via `@serwist/turbopack`, Vitest, Playwright, mise + pnpm, GitHub Actions, Vercel.

**This is plan 1 of 5.** Later plans (written after this one ships, against the real codebase): 2 — Auth & Groups (Better Auth + organization plugin, invite links, join requests, Neon setup); 3 — Meetings & Notes; 4 — Recipes & Meal Sign-ups; 5 — Prayer Bowl & My Prayers. Spec: `docs/superpowers/specs/2026-07-02-small-group-pwa-design.md`.

## Global Constraints

- Next.js 16 (latest stable), App Router, TypeScript, Turbopack.
- Node 24 and pnpm 10, pinned in `mise.toml`; every dev command is a mise task.
- Postgres 17 everywhere: Docker named volume locally, Neon in production (Neon comes in plan 2).
- Drizzle for ORM and migrations. No other DB access path.
- Tailwind CSS + shadcn/ui, mobile-first.
- Visual design is the "Hearth" theme: tokens in `docs/design/hearth/theme.css`, reference screens in `docs/design/hearth/*.dc.html` — high fidelity, colors/type/spacing/radii are final. Never hardcode a color that has a token.
- PWA via Serwist's Turbopack integration (`@serwist/turbopack`).
- Testing: Vitest (unit + integration), Playwright (e2e smoke).
- Trunk-based: `main` deploys production. Tasks 1–6 commit directly to local `main` (the GitHub repo doesn't exist until Task 7). After Task 8 enables branch protection, ALL further work — including plans 2–5 — goes through feature-branch PRs.
- Commit after every task, message style: short imperative summary.
- If a command or library API differs from what's written here (packages newer than this plan), consult that library's current docs rather than forcing the plan's version.

In `src/app/layout.tsx`, self-host the theme fonts via `next/font` (required for offline PWA; do NOT use the Google Fonts CDN `@import` from theme.css):


In `src/app/globals.css`: copy the entire `:root { … }` and `[data-theme='dark'] { … }` blocks from `docs/design/hearth/theme.css` (EXCEPT the `@import url(...)` line and the two `--font-*` lines — next/font now owns those), then map the tokens into Tailwind and shadcn's variables so utilities like `bg-background` and `text-muted-foreground` resolve to Hearth:

```css
@theme inline {
  --font-sans: var(--font-body);
  --font-serif: var(--font-heading);
}

:root {
  --background: var(--bg);
  --foreground: var(--text);
  --card: var(--surface);
  --card-foreground: var(--text);
  --primary: var(--accent);
  --primary-foreground: var(--accent-contrast);
  --muted: var(--surface-tint);
  --muted-foreground: var(--text-secondary);
  --destructive: var(--danger);
  --radius: 14px;
}
```

(Keep shadcn's other generated variables; only override the ones above. If the shadcn scaffold's variable names differ, map to its actual names — the goal is: shadcn components render in Hearth colors with no per-component overrides.)
